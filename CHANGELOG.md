# Changelog

All notable changes to this repository are documented here.

## [Unreleased]

### Added

- **Investment / Trading System (#376 — Trading 2/5: point-in-time, content-hashed market-data cache):** Introduces a durable, content-addressed cache that solves the two coupled gaps the umbrella plan #374 calls out — backtest reproducibility (providers silently revise history; reruns drift) and fetch latency (the in-memory `_adv_cache` had a 1-hour TTL and worker count was hard-coded at 5). New module `backend/agents/investment_team/market_data_cache/` writes one immutable Parquet snapshot per `(asset_class, symbol, frequency, provider, fetch_date)` under `${AGENT_CACHE}/investment_team/market_data/...` and indexes them in a new Postgres table `investment_market_data_snapshots` (the investment team's first Pattern B schema; lifespan registers it). `MarketDataCache.get_or_fetch` resolves a request as the latest snapshot covering `[start, end]` with `fetch_ts <= as_of` (None means "now"); empty-range fetches do not write a snapshot. `MarketDataService.fetch_ohlcv_range` and `fetch_multi_symbol_range` route through the cache, gain an `as_of` keyword, populate `provider_used` from both hit and miss paths, and honour `MARKET_DATA_FETCH_WORKERS` (default `min(len(symbols), 16)`); the TTL ADV cache is replaced by `MarketDataCache.derive_adv` keyed on `(per-symbol-snapshot fingerprint, lookback)` — eternally valid within a fingerprint. `compute_dataset_fingerprint` is a SHA256 over canonical bar serialization (sorted by date, `repr()` floats), order-independent across symbol set and within a symbol; `BacktestResult.dataset_fingerprint` and `PaperTradingSession.dataset_fingerprint` are populated from both data paths. Provider-driven backtests get cache awareness through a new `CachingProviderHistoricalStream` wrapper that buffers + hashes bars on the miss path, persists per-symbol snapshots on `EndOfStreamEvent`, and short-circuits to a Parquet replay on a full-symbol cache hit. Paper-trade warm-up windows persist per-symbol snapshots at cut-over and stamp the session fingerprint. `StrategyLabOrchestrator` passes `spec.audit.data_snapshot_id` as `as_of` for both strategy and benchmark fetches so a saved spec replays a consistent snapshot. New env: `INVESTMENT_MARKET_DATA_CACHE_ROOT` (operator override of cache root; falls back to `AGENT_CACHE`, then tempdir with WARN), `MARKET_DATA_FETCH_WORKERS`. New dependency: `pyarrow>=15.0,<22.0`. Code: `backend/agents/investment_team/market_data_cache/{__init__,paths,store,streaming,postgres/__init__}.py` (new), `market_data_service.py`, `models.py`, `trading_service/modes/{backtest,paper_trade}.py`, `strategy_lab/orchestrator.py`, `api/main.py` (new lifespan), `backend/agents/shared_postgres/registry.py`. Tests: `tests/test_market_data_cache.py` (12 tests) and `tests/test_dataset_fingerprint.py` (8 tests); existing `_StubMarketDataService` and the `fetch_multi_symbol_range` mock test were updated for the new `as_of` keyword and cache-routing path; the three reference-strategy golden snapshots regenerate to include the `dataset_fingerprint` field. No-op when `POSTGRES_HOST` is unset (in-memory index fallback) so unit tests don't require Postgres; production reproducibility requires Postgres.
- **Investment / Strategy Lab (#248 — pluggable execution model + fill-realism fixes):** New `ExecutionModel` protocol with two implementations behind `BacktestConfig.execution_model` (default: `realistic`). The legacy fill geometry is preserved as `OptimisticExecutionModel` (used by `tests/golden/test_simulator_invariants.py` and `tests/golden/test_reference_strategies.py`, which now opt in explicitly); selecting it elsewhere logs a one-shot warning unless `KHALA_ALLOW_OPTIMISTIC_FILLS=1`. `RealisticExecutionModel` ships three fixes: (1) **limit fills at the limit price** — a resting buy limit struck by a gap-down bar fills at `limit_price`, not the better `min(bar.open, limit_price)` (the legacy "free alpha" path); (2) **participation-cap partial fills** — orders sized above `fill_participation_cap × bar_dollar_volume` (default 10%) fill only up to the cap, with the remainder dropped (no re-quoting in v1); (3) **adverse-selection haircut on limit fills** — when `next_bar` is available, a directional toxicity term proportional to `sign(next_bar.close − bar.close) × participation_rate` widens the slippage band on the unfavourable side (capped at `adverse_selection_max_bps=50`). `SpreadPlusImpactCostModel.estimate()` now accepts a `bar_dollar_volume_usd` proxy and uses it when ADV is unavailable instead of silently defaulting market-impact to zero (logs at INFO when the fallback fires). `TradingService.run()` peeks one event ahead so the realistic model gets `next_bar` for the haircut. New `BacktestConfig.execution_model: Literal["optimistic", "realistic"]` and `fill_participation_cap: float`. Code: `backend/agents/investment_team/trading_service/engine/execution_model.py` (new), `trading_service/engine/fill_simulator.py`, `trading_service/service.py`, `execution/cost_model.py`, `models.py`. Tests: `tests/test_realistic_fill_model.py` (new, 24 tests), updates to `tests/test_cost_model.py`, golden parity pinned via explicit `execution_model="optimistic"` in `tests/golden/test_simulator_invariants.py` and `tests/golden/test_reference_strategies.py`.
- **Investment / Strategy Lab (#247 — purged walk-forward + Deflated Sharpe + composite acceptance gate):** The Strategy Lab refinement loop now evaluates strategies with López de Prado–style purged K-fold walk-forward (default K=5, embargo derived from `max(TradeRecord.hold_days)`) and replaces the legacy scalar `WINNING_THRESHOLD = 8.0` with a composite `AcceptanceGate` over four OOS criteria: Deflated Sharpe (Bailey & López de Prado, trial-counted via `ConvergenceTracker.trial_count`), IS→OOS Sharpe degradation, OOS trade count, and regime-conditional benchmark beats (≥2 of 4 VIX subwindows). Sharpe CIs come from a Politis–Romano stationary block bootstrap. New `BacktestConfig` fields (`walk_forward_enabled`, `n_folds`, `embargo_days`, `min_oos_trades`, `dsr_threshold`, `max_is_oos_degradation_pct`, `benchmark_composition`) and `BacktestResult` diagnostics (`deflated_sharpe`, `sharpe_ci_low/high`, `is_sharpe`, `oos_sharpe`, `is_oos_degradation_pct`, `oos_trade_count`, `n_trials_when_accepted`, `acceptance_reason`, `regime_results`, `fold_results`) are additive. `BacktestAnomalyDetector` downgrades `Sharpe > 5.0` from critical to warning when `dsr_aware=True` (AcceptanceGate adjudicates); a fallback re-runs anomalies with `dsr_aware=False` if walk-forward evaluation fails for a cycle. Code: `backend/agents/investment_team/strategy_lab/orchestrator.py`, `quality_gates/acceptance_gate.py`, `quality_gates/convergence_tracker.py`, `execution/walk_forward.py`, `execution/metrics.py`, `models.py`. Tests: `tests/test_walk_forward.py`, `test_deflated_sharpe.py`, `test_acceptance_gate.py`, `test_backtest_anomaly_dsr_aware.py`, `test_strategy_lab_walk_forward_integration.py`, plus the curve-fit `OverfitHardcodedDates` reference in `tests/golden/strategies.py` and its rejection snapshot in `tests/golden/test_reference_strategies.py`. Closes #247. (PRs #271, #272, #353.)
- **Nutrition / Meal Planning (SPEC-015 — pantry tracking, Phase 0):** Kicks off SPEC-015 by authoring [PLAN-015](system_design/plans/PLAN-015-nutrition-pantry.md) (work-item decomposition + rollout tracker for #220) and landing the Phase 0 foundation. New `backend/agents/nutrition_meal_planning_team/pantry/` module exports `PantryItem`, `PantryStore`, `get_pantry_store`, and `PANTRY_VERSION = "1.0.0"`; the Postgres-backed store supports add-or-increment (race-free via `INSERT ... New `backend/agents/agent_provisioning_team/tests/test_e2e_smoke.py` (gated on `KHALA_E2E=1`) drives the four-team smoke matrix (`blogging.planner`, `se.backend.api_openapi`, `planning_v3.intake`, `branding.creative_director`), proves the one-sandbox-per-agent rule with cross-team and intra-team concurrency tests, exercises the idle reaper, and asserts that `requires-live-integration` agents return 409. Companion `tests/scripts/phase6_hardening.sh` runs `docker inspect` / `ss` / `docker exec` probes against a live sandbox to confirm `ReadonlyRootfs`, `CapDrop=ALL`, `no-new-privileges`, `seccomp`, `Memory=1g`, `PidsLimit=512`, loopback-only port binding, non-root user, and pids-limit fork-bomb containment. `tests/scripts/phase6_perf_summary.py` reads the per-invoke perf log and prints cold/warm p50/p95. `backend/agents/agent_provisioning_team/sandbox/README.md` gains a "Capacity" section documenting resource caps, port binding, the cold-start observability hook, and exhaustion modes (with measured latency rows left as `TBD` for the human executor). ### Added

- **Agent Console (Phase 3 — Runs, saved inputs, diff, form editor):** Every invocation now persists to a new `agent_console_runs` Postgres table, tagged with an `author` handle derived from the shared `AuthorProfile` so data is ready for a real-auth migration. Users can save, name, describe, update, and delete per-agent inputs (`agent_console_saved_inputs`, unique on `(agent_id, name)`). The Runner picker groups golden samples and saved inputs; a new history panel (last 20 runs with click-to-reload) lives under the output pane. A `POST /api/agents/diff` endpoint returns a unified-diff string of pretty-printed sorted-key JSON between any two of `{run, saved_input, inline}` — accessible from the history row menu and the live Runner output. The input editor has a tiered Form/JSON toggle: scalars, enums, flat scalar arrays, and nested objects up to depth 2 render as typed Material inputs; unions, deep nesting, and `$ref` cycles render a JSON-textarea subtree with an "editing as JSON" chip. A background pruner trims runs to `AGENT_CONSOLE_RUNS_RETENTION` (default 200) per agent every `AGENT_CONSOLE_PRUNE_INTERVAL_S` (default 3600s). All new storage is no-op-gracefully when `POSTGRES_HOST` is unset. See [`agent_console/README.md`](backend/agents/agent_console/README.md).
- **Agent Console (Phase 2 — Runner + Sandboxes):** The Runner tab now invokes single specialist agents in warm per-team Docker sandboxes. New `backend/agents/agent_sandbox/` shells out to `docker/sandbox.compose.yml` to spin up isolated team services (dedicated `sandbox-postgres` on port 5433, isolated `khala-sandbox` network, ports 8200–8220), reuses them across invocations, and reaps idle sandboxes after `SANDBOX_IDLE_TEARDOWN_MINUTES` (default 15). New `backend/agents/shared_agent_invoke/` is a one-line-mount FastAPI shim each team's `api/main.py` includes to expose `POST /_agents/{id}/invoke`; dispatch handles plain functions, classes, factories, and coroutines with stdout/log capture. Unified API adds `POST /api/agents/{id}/invoke`, `GET/POST/DELETE /api/agents/sandboxes/{team}`, and `GET /api/agents/{id}/samples[/{name}]`. Four teams wired: blogging, software_engineering, planning_v3, branding; others appear in the catalog and get sandbox support as their `api/main.py` adds the shim. 14 manifests enriched with `inputs.schema_ref`; 12 golden sample skeletons auto-generated via `python3 -m agent_registry.scripts.generate_sample_skeletons`. Agents tagged `requires-live-integration` (e.g. `blogging.medium_stats`, `blogging.publication`) are catalogued but Runner disables the Run button with an explainer. See [`agent_sandbox/README.md`](backend/agents/agent_sandbox/README.md) and [`shared_agent_invoke/README.md`](backend/agents/shared_agent_invoke/README.md).
- **Agent Console (Phase 1 — Catalog):** New `/agent-console` page that replaces the friction-heavy `/agent-provisioning` form. Ships a browsable, searchable catalog of every specialist agent in the system with team/tag filters and a detail drawer. Backed by a new declarative registry at `backend/agents/agent_registry/` that loads per-agent YAML manifests from `backend/agents/<team>/agent_console/manifests/*.yaml` and serves them via `GET /api/agents`, `GET /api/agents/teams`, `GET /api/agents/{id}`, and `GET /api/agents/{id}/schema/{input|output}` (lazy Pydantic JSON-schema resolution). 29 seed manifests authored across `blogging`, `software_engineering` (backend_code_v2), `planning_v3`, and `branding` — remaining teams get manifested in follow-up PRs and appear automatically. The legacy `/agent-provisioning` route redirects to `/agent-console`; provisioning and environments are preserved verbatim inside the third tab. Runner tab is a placeholder pending Phase 2. See [agent_registry/README.md](backend/agents/agent_registry/README.md).
- **`llm_service` structured-output contract:** new top-level `generate_text` / `generate_structured` entrypoints and a `complete_validated` helper that layers Pydantic validation + one schema-grounded self-correction retry on top of `complete_json`. New `LLMSchemaValidationError` carries `correction_attempts_used`; `LLMJsonParseError` gained the same optional field. A CI static check (`agents/llm_service/tests/test_no_markdown_in_structured.py`) prevents `*_PROMPT` constants with Markdown/prose bodies from being routed into JSON-only methods. Legacy `complete` / `complete_text` / `complete_json` / `chat_json_round` are unchanged and remain supported. See [FEATURE_SPEC_structured_output_contract.md](backend/agents/llm_service/FEATURE_SPEC_structured_output_contract.md).

### Fixed

- **`user_agent_founder` spec generation:** the "Startup Founder Testing Persona" run no longer fails with `LLMJsonParseError` when the model returns Markdown. `FounderAgent.generate_spec` now bypasses the Strands / `chat_json_round` JSON-only transport and calls `LLMClient.complete` directly via a new `_call_text` helper. `FounderAgent.answer_question` migrated to `generate_structured` with a `FounderAnswer` Pydantic schema (canary for the new API); the bespoke regex/`json.loads` fallback was removed in favor of the self-correction guard.
- **Ollama structured JSON:** responses with unescaped double quotes inside string values (common when the model cites snippets like `"Resource": "*"`) are parsed using a **`json-repair`** fallback in `OllamaLLMClient._extract_json`. Blog copy-editor prompt updated to require escaped or single-quoted inner quotes.

### Breaking changes

- **Blogging pipeline:** `BlogReviewAgent` has been removed. The pipeline is **research → planning → writer** with a persisted `ContentPlan` (`content_plan.json` / `content_plan.md`). `POST /research-and-review` (sync and async) runs the same **research + planning** step as the full pipeline, not a separate “review” agent. `POST /full-pipeline` returns `title_choices` and `outline` derived from the approved plan; planning failure returns **422** with `planning_failed` detail. Async jobs expose a **planning** phase and optional planning observability fields on completed jobs; failed planning jobs may include `planning_failure_reason`. Optional env: `BLOG_PLANNING_MODEL`, `BLOG_PLANNING_MAX_ITERATIONS`, `BLOG_PLANNING_MAX_PARSE_RETRIES`.
